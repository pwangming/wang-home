# 计划 2.0：事件契约迁移（2.2/3 前置）

> 状态：✅ 方案已确认（2026-04-21 新增）
> 分支：`refactor/game-event-contracts`（待创建）
> 基分支：`develop`
> **硬依赖**：1.1（实时分数链路）
> 预计工作量：1 ~ 2 小时
> 模型主力：**Minimax**（前端契约改造 + 测试），Opus 做 review

---

## 📍 背景

当前代码契约仍是旧版：

- `SnakeGame` 仅 `emit('eatFood')`，无 payload
- `useGameSession.handleGameOver(finalScore, speedMult, scoreMult)` 仅 3 参
- `GameView` 直接把 `@eat-food` 绑定到 `playEatSound`

而后续 2.2（食物变化）和 3（成就系统）都依赖新契约：

- `emit('eatFood', { type, score })`
- `handleGameOver(finalScore, speedMult, scoreMult, gameContext)`

若不先迁移，会导致 2.2/3 同时改接口、冲突和回归风险明显上升。

---

## 🎯 目标

1. 在不改变玩法的前提下完成事件签名迁移
2. 给 2.2/3 提供稳定接口，不在功能 PR 里再改函数签名
3. 补齐测试，保证契约升级可回归验证

---

## 📦 改动清单

### 1. `client/src/components/game/SnakeGame.vue`

- `defineEmits` 增加 `scoreUpdate`
- 吃食物时改为：

```js
emit('eatFood', { type: 'normal', score: Math.round(1 * props.scoreMultiplier) })
```

- `watch(score, ...)` 实时 emit `scoreUpdate`

> 本计划仅迁移契约，不引入多食物逻辑。

### 2. `client/src/views/GameView.vue`

- `@eat-food` 从直接播音改为 `onEatFood(payload)`
- 新增 `onScoreUpdate(score)` 对接 `@score-update`
- `onEatFood` 内部做两件事：
  - `playEatSound()`
  - 累加基础计数器（至少 `foodEaten`）
- `@game-over` 改为走包装函数，构造 `gameContext` 后调用 composable

### 3. `client/src/composables/useGameSession.js`

- `handleGameOver` 签名扩展为：

```js
async function handleGameOver(finalScore, speedMult, scoreMult, gameContext = null)
```

- 本迭代中 `gameContext` 仅透传/保留，不触发成就接口

### 4. 悬空调用治理（同 PR 完成）

当前 `api.leaderboard.startSession('/game-sessions/start')` 在后端未实现（grep 确认 `server/src/` 无对应路由），前端静默失败。

**决策**：✅ 移除前端调用（方案 2）

**理由**：
- 后端该路由无当前需求（游戏会话开始无需服务端确认）
- 实现后端会引入新的 RLS / 限流面，超出"契约迁移"范畴
- 如后续需要（防作弊、会话审计），另起独立计划

**改动点**：
- `client/src/composables/useGameSession.js:37` — 删除 `await api.leaderboard.startSession(selectedSpeed.value)` 调用
- `client/src/lib/api.js:80-81` — 删除 `startSession` 方法
- 对应测试同步清理
