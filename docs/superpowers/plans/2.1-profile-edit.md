## 💡 方案设计（决策已确认）

### 决策 1：改密码是否要求"当前密码"？ ✅ 要求

- 理由：防 session 被劫持后被直接改密锁号
- 前端表单：旧密码 + 新密码 + 确认新密码
- 后端：先用旧密码调 `signInWithPassword` 验证，通过再调 `updateUser({ password })`

### 决策 2：改邮箱流程？ ✅ Supabase 默认流程

Supabase `updateUser({ email })` 行为：
- 会向**新邮箱**发送确认邮件
- 用户点击链接后才真正生效（旧邮箱仍可用直到生效）
- 复用已有 magic link callback 基础设施（`POST /api/auth/callback`）处理邮箱变更确认

**MVP 采用默认流程。** 向旧邮箱发告警（Supabase "secure email change"）留作后续加固，部署前在 Dashboard 检查开关。

### 决策 3：改邮箱是否要求当前密码？ ✅ 要求

- 理由：同决策 1，防 session 劫持后邮箱被换

### 决策 4：Rate limiting？ ✅ 确认数值

- `/api/auth/update-password`：**10 次 / 15min / user**
- `/api/auth/update-email`：**5 次 / 60min / user**（邮件型更严）

### 前端：`client/src/composables/useAuthCallback.js` — 多 type 分支（**必做，2.1a**）

**现状问题**：当前只认 `type=signup`（`useAuthCallback.js:4`），Supabase 改邮箱会发 `type=email_change` magic link，到达前端后会被默默丢弃。

**改动**：

```js
export function useAuthCallback({ api, authStore, router, message }) {
  async function handleCallback() {
    const hash = window.location.hash
    const params = new URLSearchParams(hash.substring(1))
    const type = params.get('type')
    const accessToken = params.get('access_token')
    const refreshToken = params.get('refresh_token')

    if (!accessToken) return

    // 支持的 type 白名单
    const SUPPORTED = ['signup', 'email_change', 'recovery']
    if (!SUPPORTED.includes(type)) return

    try {
      const result = await api.auth.callback(accessToken, refreshToken)
      authStore.user = result.user
      authStore._startHeartbeat()
      history.replaceState(null, '', window.location.pathname)

      // 按 type 显示不同反馈
      const msg = {
        signup: '邮箱验证成功！欢迎加入',
        email_change: '邮箱已更新为新地址',
        recovery: '已登录，请尽快修改密码'
      }[type]
      message.success(msg)

      // recovery type 额外跳到修改密码页
      if (type === 'recovery') router.push('/reset-password?step=set-new')
    } catch {
      history.replaceState(null, '', window.location.pathname)
      message.error('邮箱链接验证失败，请重试')
      router.push('/login')
    }
  }

  return { handleCallback }
}
```

**后端 `/api/auth/callback` 是否需改？** ——不需要。后端只做 `supabase.auth.getUser(accessToken)` + session 写入，对 type 不关心。前端按 type 分派反馈 UI 即可。

### callback 执行入口治理（新增，2.1a）

当前 `handleCallback()` 在 `GameView` 执行，导致用户在非 `/game` 路由访问 magic link 时流程不稳定。改为全局入口：

- 方案 A（推荐）：`App.vue` 挂载时执行一次 callback 检查
- 方案 B：路由 `beforeEach` 钩子检测 hash 并分流

要求：`email_change` 与 `recovery` 不依赖进入游戏页。
