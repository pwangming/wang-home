## 🚨 待修复的问题

### 🔴 HIGH-1：`/reset-request` 缺 rate limiter

**证据**：
- `server/src/routes/auth.js:129` `router.post('/reset-request', createCsrfMiddleware(), async ...)` — 只有 CSRF
- `server/src/routes/auth.js:4` 已导入 `createCallbackRateLimiter` 等，但 reset 相关路由未使用

**风险**：攻击者可对任意邮箱（或一批邮箱）发起无限次重置邮件请求，造成骚扰 + Supabase 邮件配额耗尽。

**现有参考**：`/callback` 路由（auth.js:219）已用 `createCallbackRateLimiter()`，格式可直接复用。

### 🔴 HIGH-2：`/reset-confirm` 缺 rate limiter

**证据**：`server/src/routes/auth.js:149` 只有 CSRF。

**风险**：token 暴力枚举（Supabase token 虽然熵高，但 defense-in-depth 是标准做法）。

### 🔴 HIGH-3：`supabase.auth.updateUser(token, { password })` 调用签名已确认错误

**证据**：`auth.js:162`
```js
const { data, error } = await supabase.auth.updateUser(token, { password })
```

**确认结果**：本地 `@supabase/auth-js` 类型定义明确 `updateUser(attributes, options?)`，即**只接受 attributes 对象**。当前 `updateUser(token, { password })` 为错误调用。

**为什么测试没发现**：`server/tests/routes/auth.test.js:328` 的 mock 是按代码调用签名写的（`mockAuth.updateUser.mockResolvedValue(...)`），所以测试通过但真实 SDK 可能失败。

**需要做的事**：
1. 先用 recovery token 建立临时 session（`setSession`）
2. 再调 `updateUser({ password })`
3. 修复 mock 测试签名并做真实流程验证

## 📦 改动清单

### 1. `server/src/middleware/rateLimit.js`

新增两个 rate limiter：

```js
export function createResetRequestRateLimiter() {
  return createRateLimiter({
    windowMs: 60 * 60 * 1000,  // 1 hour
    maxRequests: 3,            // 1 小时最多 3 次发送
    keyGenerator: (ctx) => `reset-request:${ctx.request.body?.email || ctx.ip}`
  })
}

export function createResetConfirmRateLimiter() {
  return createRateLimiter({
    windowMs: 15 * 60 * 1000,  // 15 minutes
    maxRequests: 10,           // 和 callback 一致
    keyGenerator: (ctx) => `reset-confirm:${ctx.ip}`
  })
}
```

**设计说明：**
- reset-request 用 email 做 key（而非 IP），防止同一邮箱被骚扰；限额 1 小时 3 次
- reset-confirm 用 IP 做 key（token 在 body，用 IP 更稳），15 分钟 10 次

### 2. `server/src/routes/auth.js`

**导入更新**（第 4 行附近）：

### 4. `server/tests/routes/auth.test.js`

新增测试用例（附加在现有 `describe('POST /api/auth/reset-request')` 块后）：

```js
describe('POST /api/auth/reset-request rate limiting', () => {
  test('returns 429 after exceeding 3 requests in 1h for same email', async () => {
    mockAuth.resetPasswordForEmail.mockResolvedValue({ error: null })
    for (let i = 0; i < 3; i++) {
      await simulateRequest(app, 'POST', '/api/auth/reset-request', { email: 'test@t.com' })
    }
    const res = await simulateRequest(app, 'POST', '/api/auth/reset-request', { email: 'test@t.com' })
    expect(res.status).toBe(429)
  })
})

describe('POST /api/auth/reset-confirm rate limiting', () => {
  test('returns 429 after exceeding 10 requests in 15min from same IP', async () => {
    mockAuth.updateUser.mockResolvedValue({ data: null, error: { message: 'Token expired' } })
    for (let i = 0; i < 10; i++) {
      await simulateRequest(app, 'POST', '/api/auth/reset-confirm', { token: 'x', password: 'abcdef' })
    }
    const res = await simulateRequest(app, 'POST', '/api/auth/reset-confirm', { token: 'x', password: 'abcdef' })
    expect(res.status).toBe(429)
  })
})
```

**注意**：rate limiter 是单例，测试间可能互相污染。**已确定解决方案**：

在 `server/src/middleware/rateLimit.js` 中导出一个 `__resetAll()`（仅 `NODE_ENV === 'test'` 时可用）：

```js
const _stores = []  // 把每个 factory 创建的 store 推进来

export function __resetAll() {
  if (process.env.NODE_ENV !== 'test') return
  _stores.forEach(s => s.clear())
}
```

测试里在每个 rate limit 相关 `describe` 的 `beforeEach` 调用 `__resetAll()`，避免共享状态。

（等价方案：每个 describe 新建 app 实例。选 `__resetAll` 更轻量。）

## ⚠️ 风险与缓解

| 风险 | 缓解 |
|---|---|
| Rate limit 单例在测试间污染 | 每个测试 block 重新 setup app / 或导出 reset 方法 |
| mock 仍沿用错误签名导致“假通过” | 先修 mock 再修实现，确保失败可见 |
| CSRF 先于 rate limit 顺序错误导致 CSRF 绕过被计数 | 保持 `csrf → rateLimit` 顺序（现在顺序已是这样） |
| 真实 Supabase 测试需要测试账号 | 使用 dev Supabase 项目 + 一次性邮箱 |
| rate limit 用 email 作 key 可被填写他人邮箱骚扰计数 | 实际上这反而是防骚扰：每个邮箱独立计数，不会拖累其他人 |
