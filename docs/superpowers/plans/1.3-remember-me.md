## 💡 方案决策（需 Opus 定论的点）

### 决策 1：maxAge 数值（已定）

- **勾选"记住我"**：`30 天` = `30 * 24 * 60 * 60 * 1000 = 2,592,000,000 ms`
  - 行业常见做法（GitHub、Discord 用的都在 14~30 天范围）
- **不勾选**：采用 **方案 A** `ctx.session.maxAge = 'session'`
  - 本地 `koa-session` 源码与类型定义均支持 `'session'`
  - fallback 方案 B（2 小时）保留为应急回退，不作为默认实现

**现状参考**：`server/src/index.js:26-31` 的 `sessConfig` 当前**没有** `maxAge` 字段，走 koa-session 默认值（通常是 24h），和计划假设不一致。

### 决策 2：Supabase refresh token 策略

- Supabase access token 通常 1 小时过期，refresh token 更长
- 当前后端保存了 `ctx.session.supabaseAccessToken` 和 `ctx.session.supabaseRefreshToken`
- 30 天内 session 有效，但 access token 过期后怎么办？
  - **选项 A**：现状延续——依赖 authMiddleware 出错时提示重登
  - **选项 B**：加 refresh 逻辑（access token 过期自动刷新）
- **推荐 A**：本计划不引入 refresh 复杂度，后续独立迭代处理

### 决策 3：rememberMe 参数位置

- body 传 `{ email, password, rememberMe: true }`
- 而非放 header/query — body 最自然

### 决策 4：安全考量

- rememberMe 不影响 `httpOnly` / `sameSite: lax` / `secure`（这些保持）
- 共享电脑用户可用不勾选 → 关闭浏览器 session 即失效
- 勾选的长 session 仍受限于 CSRF token 保护

## 📦 改动清单

### 1. `server/src/routes/auth.js` — login 路由

第 75-112 行附近：

```js
router.post('/login',
  createCsrfMiddleware(),
  createLoginRateLimiter(),
  async (ctx) => {
    const { email, password, rememberMe } = ctx.request.body  // ← 新增

    if (!email || !password) {
      fail(ctx, 400, 'email and password are required')
      return
    }

    const { data, error } = await supabase.auth.signInWithPassword({ email, password })

    if (error) {
      fail(ctx, 401, error.message)
      return
    }

    // Set session cookie with maxAge based on rememberMe
    if (ctx.session) {
      ctx.session.supabaseAccessToken = data.session.access_token
      ctx.session.supabaseRefreshToken = data.session.refresh_token
      ctx.session.userId = data.user.id

      // ← 新增：根据 rememberMe 动态设置 maxAge
      ctx.session.maxAge = rememberMe
        ? 30 * 24 * 60 * 60 * 1000   // 30 days
        : 'session'                   // browser-session cookie
    }

    const userScopedClient = createUserScopedClient(data.session.access_token)
    const { data: profileData } = await userScopedClient
      .from('profiles')
      .select('username')
      .eq('id', data.user.id)
      .maybeSingle()

    ok(ctx, { user: { id: data.user.id, email: data.user.email, username: profileData?.username || null } })
  }
)
```

### 5.（可选）`server/src/index.js` — 全局默认 maxAge

第 26-31 行的 `sessConfig` 当前没有 `maxAge`。可以显式写一个默认值，使未勾"记住我"走 register / callback 等路径也有一致行为：

```js
const sessConfig = {
  key: 'session',
  httpOnly: true,
  sameSite: 'lax',
  secure: process.env.NODE_ENV === 'production',
  maxAge: 'session'  // default: browser-session cookie
}
```

**但**：如果设置 `maxAge: 'session'` 作全局默认，会影响 `/register` 自动登录后的行为。这里需要 Opus 判断是否安全（共享电脑注册完浏览器关了就登出——其实用户体验还挺合理）。

## ⚠️ 风险与缓解

| 风险 | 缓解 |
|---|---|
| koa-session 对 `maxAge='session'` 的行为与预期不符 | 本地验证 + 查 koa-session 源码 / 文档 |
| access token 在 30 天 session 中过期，API 500 | 现状已有：authMiddleware 检测失败会清 session。接受此行为，refresh 留后续迭代 |
| 共享电脑用户误勾"记住我" | UI 层可加提示（超出本计划范围） |
| 旧 session（无 maxAge）的用户升级后行为 | koa-session 会沿用现有 cookie maxAge，新登录才生效，平稳过渡 |
| `rememberMe: undefined` 的老请求 | Falsy 值一律走 `'session'`，兼容 |
