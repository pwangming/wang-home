# AI 协作

本文档记录 Claude Code 与 Codex 在本项目中的分工、交接和审查流程。`AGENTS.md` 仍是主规则源；如本文档与 `AGENTS.md` 冲突，以 `AGENTS.md` 为准。

## 协作目标

- 让 Claude Code 的深度规划能力和 Codex 的落地执行能力互相补充。
- 降低大功能、支付、数据库、部署和重构中的返工风险。
- 避免两个 AI 因规则不一致、上下文不一致或分支混乱而互相覆盖。
- 让每次交接都能被审查、复现和回滚。

## 角色分工

### Claude Code

主要负责：

- 架构设计和方案推演。
- 需求拆解和风险识别。
- 大功能开始前的选型、边界和取舍分析。
- 为 Codex 提供实施前的设计输入。

默认协作模式：

- 优先输出方案交给 Codex 落地。
- 用户明确指定 Claude Code 直接执行时，按 `AGENTS.md` "功能与变更准入标准"走流程：
  - 低风险（文档、注释、小改）直接执行。
  - 中/高风险先方案、等用户确认、再执行。
- 不另设"Claude Code 必须只做方案不改代码"硬性限制；具体不可直接执行的类别见下方"高风险协作边界"节。

### Codex

主要负责：

- 审查 Claude Code 的方案是否适合当前仓库。
- 将已确认方案拆成可执行的小步骤。
- 落地代码、文档、测试和验证。
- 维护 `AGENTS.md`、`CLAUDE.md` 和 `docs/*`。
- 运行本地检查、测试、构建和实际页面验证。

Codex 不应直接照单全收 Claude Code 的方案；落地前必须检查方案是否符合当前代码结构、安全边界、分支流程、测试策略和项目约束。

## 推荐协作流程

```text
用户提出问题/功能
-> Claude Code 输出方案草案
-> 用户把方案交给 Codex 审查
-> Codex 指出风险、过度设计、缺口和拆分建议
-> 用户确认执行范围
-> Codex 从 develop 切任务分支并落地
-> Codex 运行测试/验证并汇报
-> PR 合入 develop 后用 Vercel 预览验证
```

对于低风险、小范围、用户明确要求直接修改的任务，可以跳过 Claude Code 方案阶段，由 Codex 直接执行。

## Claude Code 方案模板

Claude Code 输出架构或功能方案时，优先使用以下结构：

```text
目标：
背景问题：
推荐方案：
备选方案：
涉及模块：
数据模型/API 变化：
迁移步骤：
测试策略：
风险点：
需要 Codex 审查的问题：
```

## Codex 审查模板

Codex 审查 Claude Code 方案时，优先回答：

```text
结论：
是否符合当前仓库：
是否过度设计：
主要风险：
需要补充的信息：
建议拆分：
建议先做：
建议暂缓：
需要用户确认：
```

审查重点：

- 是否符合 `AGENTS.md` 和当前有效 `docs/*`。
- 是否符合现有代码结构和技术栈。
- 是否引入不必要的新依赖、新抽象或大范围重构。
- 是否影响认证、session、cookie、CSRF、数据库、支付、部署或用户数据。
- 是否能拆成小分支、小 PR。
- 是否有测试策略、实际页面验证和回滚方式。
- 是否遗漏环境变量、迁移、幂等、错误处理或安全审查。

## 审查/建议提案模板

当 Claude Code 或 Codex 对仓库现状做审查、给出修订建议、或拆条目让用户逐项确认时，优先使用以下结构：

```text
## 提案 #N：<标题>

### 问题
<事实描述：当前规则 / 代码 / 文档存在什么问题，引用具体文件:行号>

### 建议
<推荐改法。代码/规则可贴片段>

### 优缺点
| 方案 | 优 | 缺 |
|---|---|---|
| 推荐方案 | ... | ... |
| 备选 1 | ... | ... |
| 备选 2 | ... | ... |

### 最佳实践（取舍原因）
<为什么选推荐方案：原则、先例、可维护性、与项目现状的契合度>

### 具体改法（如需立即落地）
<可贴最终内容或 diff>

确认后执行。
```

适用场景：

- 大批量审查（AGENTS.md / docs 体系审查）拆成多条提案逐一发给用户确认。
- 单条规则、单段代码、单个流程的修订建议。
- 与"Claude Code 方案模板"区分：方案模板用于**新功能/架构设计**；提案模板用于**审查/修订/优化**。

## Codex 执行计划模板

方案通过审查并准备落地时，Codex 应给出执行版计划：

```text
本次只做什么：
明确不做什么：
预计修改文件/模块：
新增或调整的测试：
实际页面验证：
验收标准：
回滚方式：
未决问题：
```

中风险和高风险任务必须等待用户确认执行版计划后再改代码。

## 交接格式

用户从 Claude Code 交接方案给 Codex 时，建议附上：

```text
这是 Claude Code 的方案，请先审查，不要直接改代码。

重点看：
- 是否适合当前仓库
- 是否过度设计
- 能否拆成小步
- 是否有安全/测试/部署风险
- 哪些应先做，哪些应暂缓
```

如果已经确认要执行，可以写：

```text
按审查后的最小方案执行。
```

## 分支与工作区

- 同一时间只允许一个 AI 作为主驾驶修改仓库。
- Claude Code 默认不创建代码分支；Codex 负责从 `develop` 切 `codex/<type>-<task>` 分支执行。
- 不让两个 AI 同时在同一分支或同一批文件上做大范围修改。
- Codex 开始执行前必须检查 `git status --short --branch`。
- 如果发现既有未提交改动，必须判断是否与任务相关；不得覆盖、还原或顺手提交用户已有改动。

## 高风险协作边界

以下任务即使已有 Claude Code 方案，也必须由 Codex 再次审查并等待用户确认：

- 认证、授权、session、cookie、CSRF。
- 数据库 schema、RLS、迁移、seed、生产数据。
- 支付、订单、退款、webhook、外部回调。
- 部署配置、环境变量、密钥、生产服务。
- GitHub release/merge、`main`、`release/*` 相关操作。
- 大范围重构或架构调整。
- MCP、plugins、skills 或用户级配置修改。

## 冲突处理

当 Claude Code 方案、Codex 判断、用户要求或文档规则冲突时：

- 以 `AGENTS.md` 为最高优先级。
- 若 `docs/*` 与 `AGENTS.md` 冲突，以 `AGENTS.md` 为准，并指出需要修正文档。
- 若 Claude Code 方案与当前代码不符，Codex 必须直接指出并给出替代方案。
- 若用户要求存在安全、维护性或流程风险，Agent 必须直接说明，不默认附和。
- 若无法判断当前方案是否仍有效，先回到用户确认，不猜测执行。
- 执行中发现规范冲突或执行偏差时，按 `AGENTS.md` "规范冲突与偏差处理"节立即提醒用户，并确认是修规范、修执行逻辑，还是记录临时例外。

## 结束汇报

Codex 完成实现或文档修改后，按 `docs/development-workflow.md` §8 通用汇报模板说明。

如果任务只做方案审查（无代码改动），按以下结构汇报：

- 是否建议执行。
- 建议执行的最小范围。
- 需要暂缓或重新设计的部分。
