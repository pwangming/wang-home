# 安全事件复盘模板

本目录用于记录已经处理完的安全事件、生产事故或高风险异常复盘。不要在复盘中写入真实密钥、token、连接串、完整用户个人信息或可复现攻击细节；必要时使用脱敏 ID 与占位符。

## 文件命名

```text
YYYY-MM-DD-short-title.md
```

示例：

```text
2026-04-30-auth-callback-cookie-issue.md
```

## 模板

```markdown
# <事件标题>

## 摘要

- 日期：
- 影响范围：
- 严重等级：CRITICAL / HIGH / MEDIUM / LOW
- 当前状态：已缓解 / 已修复 / 观察中
- 负责人：

## 时间线

| 时间 | 事件 |
|---|---|
| YYYY-MM-DD HH:mm | 发现问题 |
| YYYY-MM-DD HH:mm | 缓解措施 |
| YYYY-MM-DD HH:mm | 完成修复 |

## 影响

- 受影响功能：
- 受影响用户或数据范围：
- 是否涉及认证、数据库、支付、密钥或用户数据：
- 是否需要通知用户或平台：

## 根因

描述技术根因、流程根因和未及时发现的原因。避免记录可直接复现攻击的敏感细节。

## 处置

- 已执行的缓解措施：
- 已合并的修复：
- 已轮换或废弃的密钥 / token / webhook（如适用，使用占位符）：
- 已恢复的服务或数据：

## 验证

- 本地验证：
- staging / preview 验证：
- production 验证：
- 安全审查结果：

## 后续行动

| 动作 | 负责人 | 截止时间 | 状态 |
|---|---|---|---|
|  |  |  | TODO |

## 剩余风险

- 待补充
```

## 处理规则

- 复盘可以引用 PR、commit、workflow run、部署记录，但不要粘贴敏感日志全文。
- 涉及用户数据泄漏、支付、生产密钥或外部平台滥用时，先按 `docs/security-boundaries.md` 停止写操作、评估影响、轮换密钥，再补复盘。
- 复盘中的长期改进项应拆成独立 issue / plan，不在 incident 文档里直接展开大方案。
